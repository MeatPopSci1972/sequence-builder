## What this project is

Single-file, zero-dependency static HTML application for building UML sequence diagrams visually. Outputs PlantUML or Mermaid syntax. Runs entirely in the browser — no server, no npm, no build pipeline except the one-step `node build.js` that syncs the store module into the HTML.

**Deployment:** `git push` → GitHub Pages serves `sequence-builder.html` directly. Done.

## Where we left off (v0.9.16)

### What was built in this session (2026-03-17)

**v0.9.10 — Debug console cleanup**  
`let DEBUG_VERBOSE = false` gates three high-frequency log calls (`msg:updated`, `save`, `msg-dblclick`). Verbose toggle button in debug toolbar flips it at runtime. All other `dbg()` calls stay unconditional — they're one-per-gesture.

**v0.9.11 — Double-dispatch on message click fixed**  
`setSelected(msg, 'message', true)` in the kind-cycle timer was triggering the `associateOnClick` fallback, firing `UPDATE_MESSAGE` twice per click. Fixed by passing `false`.

**v0.9.12 — Two deeper message interaction bugs fixed**  
1. `_wrapSelected` was calling `Object.freeze(obj)` on the live store object — silently blocking all future `UPDATE_MESSAGE` mutations on selected messages. Fixed: `Object.freeze({ ...obj })` freezes a copy, not the original.  
2. Slow dblclick (>220ms between clicks) caused both kind-cycle and direction-cycle to fire. Fixed with `_msgDblclickPending` flag.

**v0.9.13 — Click-to-cycle-kind removed (Option A)**  
Single-click on a message now selects it and opens Properties. Direction cycles on dblclick. Kind changes via Properties panel only. Eliminated `_msgClickTimer`, `_msgDblclickPending`, the `setTimeout(render,0)` deferral, and all click/dblclick race conditions. Net: −19 lines.

**v0.9.14 — Fragment resize (SE corner)**  
Resize handle on selected fragments. `dragging._resizeHandle = 'se'` during resize; mouseup commits one undoable `RESIZE_FRAGMENT`. Min size 60×40.

**v0.9.14 — Fragment z-order fixed**  
Layer order was `fragments → lifelines → messages → notes → actors`. Fixed to `lifelines → fragments → messages → notes → actors`. Selected fragment renders last within its layer (paints on top when fragments overlap).

**v0.9.15 — Place-actor mode**  
`btn-add-actor` and `A` key now toggle a placement mode (same pattern as connect mode). Ghost actor follows cursor. Click canvas → places actor at that X. `ADD_ACTOR` store handler now accepts `payload.x` with `?? getNextActorX()` fallback. 2 new tests (31 total).

**v0.9.16 — Actor drag reflow**  
Dragging an actor between two others pushes everything into clean evenly-spaced positions on drop. New `REFLOW_ACTORS` store action — one snapshot, one Ctrl+Z restores all actors. 2 new tests (33 total).

## Architecture in one diagram

```
User event
    │
    ▼
store.dispatch(action)          ← ONLY mutation entry point
    │
    ├── log.push(action)
    ├── pushSnapshot()           ← if meta.undoable !== false
    ├── handler mutates state    ← try/catch: pops log on throw
    └── emit(event, payload)
              │
              ▼
        store.on() listeners     ← registered in HTML, never in store
              │
              ├── update uiState
              └── render() or requestRender()
                      │
                      ├── render()        — mutations (immediate)
                      └── requestRender() — drag (rAF-batched)
                              │
                              ▼
                          SVG layers (inside <g id="viewport">)
                          lifelines → fragments → messages → notes → actors
```

**Invariant:** `store` never reads `uiState`. `store` never calls `render()`.

## State model

```js
// store.state — serializable, persistable, narrow
{
  actors:    [{ id, x, label, type, emoji? }],
  messages:  [{ id, fromId, toId, label, kind, direction, y,
                protocol?, port?, auth?, dataClass? }],
  notes:     [{ id, x, y, text }],
  fragments: [{ id, x, y, w, h, kind, cond }],
  nextId:    Number
}

// uiState — never persisted, resets on reload
{
  selected:       null | { ...obj, _type, _ref, _preview? },
  mode:           'select' | 'connect' | 'place-actor',
  connectFrom:    null | actorId,
  pendingActorId: null | actorId,
  format:         'plantuml' | 'mermaid',
  placeGhostX:    null | Number   // transient — place-actor mode only
}
```

**Persistence rule:** if it survives a browser refresh → `store.state`. If it resets on reload → `uiState`.

## The five things most likely to confuse a fresh AI instance

**1. Never edit the `@@STORE-START…@@STORE-END` block in the HTML.**  
It's generated. Edit `sequence-builder.store.js`, run tests, run `node build.js`.

**2. `_wrapSelected` freezes a COPY, not the live store object.**  
`Object.freeze({ ...obj })` — the original stays mutable for dispatch. Freezing `obj` directly silently blocks all future mutations on that object in non-strict script context.

**3. `render()` ≠ `requestRender()`.**  
`render()` is immediate — use after mutations. `requestRender()` is rAF-batched — use during drag. Never swap them.

**4. `const state = _store.state` is a live alias, not a copy.**  
Reading it is fine everywhere. Writing to it directly (`state.actors.push(...)`) bypasses the log and undo won't see it.

**5. `REFLOW_ACTORS` exists for batch actor moves.**  
Dispatching N individual `MOVE_ACTOR` actions pushes N snapshots. Use `REFLOW_ACTORS` with `payload.positions: [{id, x}]` when multiple actors need to move in one undoable step.

## Key settled decisions (do not relitigate without updating the main handoff)

- **Narrow store** — `store.state` is the diagram record only. `uiState` is separate.
- **Synchronous dispatch** — no async in the dispatch path, ever.
- **`REFLOW_ACTORS` for batch moves** — one snapshot covers all actor positions.
- **Click on message = select only** — kind changes via Properties panel. dblclick cycles direction. This is final (Option A decision in v0.9.13).
- **`save` fires on every `render()`** — save-on-render, not save-on-dispatch. Acceptable at current diagram sizes.
- **Fragment layer order** — lifelines first, then fragments, then everything else on top.
- **`_ref` is a frozen copy** — `_wrapSelected` freezes `{ ...obj }`, not `obj` itself.
