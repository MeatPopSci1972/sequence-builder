## What This Is

Single-file, zero-dependency static HTML application for building UML sequence diagrams visually. Outputs PlantUML or Mermaid syntax. Runs entirely in the browser — no server, no npm. Drop `sequence-builder.html` onto GitHub Pages, Netlify, S3, or open directly in a browser.

**Deployment:** `git push` → GitHub Pages serves `sequence-builder.html` directly. No CI required.

## Architecture

### Core Separation (v0.6.0, unchanged)

```
store.state   — diagram data only: actors, messages, notes, fragments, nextId
               Serializable. Persistable. Never touches DOM.

uiState       — UI-only, never persisted:
               selected, mode, connectFrom, pendingActorId, format, placeGhostX
               Resets on reload by definition.
```

**Persistence rule:** if it survives a browser refresh → `store.state`. If it resets on reload → `uiState`. This rule resolves every future boundary question without debate.

### One-Way Data Flow

```
User event
    │
    ▼
store.dispatch(action)          ← ONLY mutation entry point
    │
    ├── log.push(action)
    ├── pushSnapshot()           ← if meta.undoable !== false
    ├── handler mutates state    ← try/catch: pops log on throw
    └── emit(event, payload)
              │
              ▼
        store.on() listeners     ← registered in HTML, never in store
              │
              ├── update uiState (selected, mode, etc.)
              ├── call setSelected() for selection side-effects
              │     └── pushes announcement to #select-announce live region
              └── render() or requestRender()
                      │
                      ├── render()        — mutations (immediate)
                      └── requestRender() — drag (rAF-batched)
                              │
                              ▼
                          SVG layers (inside <g id="viewport">)
                          lifelines → fragments → messages → notes → actors
```

**Invariant:** `store` never reads `uiState`. `store` never calls `render()`.

### Data Model

```js
// store.state — serializable, persistable, narrow
{
  actors:    [{ id, x, label, type, emoji? }],
  messages:  [{ id, fromId, toId, label, kind, direction, y,
                protocol?, port?, auth?, dataClass? }],
  notes:     [{ id, x, y, text }],
  fragments: [{ id, x, y, w, h, kind, cond }],
  nextId:    Number   // monotonic counter, never reset mid-session
}

// uiState — never persisted, resets on reload
{
  selected:       null | { ...obj, _type, _ref, _preview? },
  mode:           'select' | 'connect' | 'place-actor',
  connectFrom:    null | actorId,
  pendingActorId: null | actorId,
  format:         'plantuml' | 'mermaid',
  placeGhostX:    null | Number   // transient — place-actor mode only
}
```

### rAF Render Batching

```js
let _rafPending = false;
function requestRender() {
  if (_rafPending) return;
  _rafPending = true;
  requestAnimationFrame(() => { _rafPending = false; render(); });
}
```

`*:moved` store events use `requestRender()`. All other events call `render()` directly. These two paths must not be collapsed — direct `render()` on mutations is correct behavior, not a performance oversight.

### Property Visitor Pattern (v0.7.0)

`renderProperties()` and `applyPropertyChange()` are both driven by a single `propertyVisitors` object keyed by `_type`:

```js
propertyVisitors = {
  actor:    { html(s), apply(s), announce(s) },
  message:  { html(s), apply(s), announce(s), afterRender(s) },
  note:     { html(s), apply(s), announce(s) },
  fragment: { html(s), apply(s), announce(s) },
}
```

**To add a new element type:** register one entry in `propertyVisitors`. No changes needed to `renderProperties`, `applyPropertyChange`, or `setSelected`.

### Zoom System (v0.7.0)

All SVG layers are wrapped in `<g id="viewport">`. `applyZoom(z)` sets a `scale(z)` transform on that group and updates the `_zoom` variable.

```js
let _zoom = 1.0;  // range: 0.25–3.0, step 0.1
```

All `mousedown` and `mousemove` coordinate calculations divide by `_zoom` before computing diagram-space positions — without this, drag targets drift at non-100% zoom.

Triggered by: toolbar Zoom+/Zoom− buttons, the 100% reset button, Ctrl+wheel on the canvas, and Ctrl++/−/0 keyboard shortcuts.

## The Five Things Most Likely to Confuse a Fresh AI Instance

**1. Never edit `@@STORE-START…@@STORE-END` in the HTML.**
It's generated. Edit `sequence-builder.store.js`, run tests, run `node build.js`. The build applies a rename pass for constants that collide with app-level names (`ACTOR_W`→`STORE_ACTOR_W`, etc.) — those renames live in the `RENAMES` array in `build.js`. If store.js gains new module-level constants that collide, add them there.

**2. `_wrapSelected` freezes a COPY, not the live store object.**
`Object.freeze({ ...obj })` — the original stays mutable for dispatch. Freezing `obj` directly silently blocks all future mutations on that object in non-strict script context. Freeze is applied only when `obj._preview` is falsy — preview scratch objects must stay writable because the palette form harvests values from them before calling `dispatch`.

**3. `render()` ≠ `requestRender()`.**
`render()` is immediate — use after mutations. `requestRender()` is rAF-batched — use during drag. Never swap them. `_saveDiagram()` is called at the end of `render()` — it will not fire on drag frames, only on committed mutations.

**4. `const state = _store.state` is a live alias, not a copy.**
Reading it is safe everywhere. Writing to it directly (`state.actors.push(...)`) bypasses the log and undo won't see it. `uiState.selected._ref` is also a live pointer into the store — the visitor `apply(s)` methods all go through `dispatch(UPDATE_*)` correctly, but direct writes to `_ref` are the most dangerous pattern in the codebase.

**5. `REFLOW_ACTORS` exists for batch actor moves.**
Dispatching N individual `MOVE_ACTOR` actions pushes N snapshots. Use `REFLOW_ACTORS` with `payload.positions: [{id, x}]` when multiple actors need to move in one undoable step. All mid-drag `MOVE_ACTOR` dispatches are `undoable: false` — only the final drop commit is undoable.

## Settled Design Decisions (Do Not Relitigate)

**Narrow store.** `store.state` is the diagram record only. `uiState` is a separate explicit object. The boundary: "does this survive a browser refresh?" Yes → store. No → uiState.

**Synchronous dispatch.** No async in the dispatch path, ever. The only async is the custom actor modal (user input) and the Ollama API call (network). Both are out-of-band.

**`requestRender()` for drag, `render()` for mutations.** These are intentionally distinct and must not be collapsed.

**Operation-level commands, not field-level.** `UPDATE_MESSAGE` carries all changed fields in one dispatch. The caller computes the next value. Prevents the undo stack from recording every keystroke.

**`meta.undoable: false` for non-undoable actions.** Log everything, filter on undo scan. Mid-drag dispatches are non-undoable — the whole drag gesture is one undo step.

**`REFLOW_ACTORS` for batch moves.** One snapshot covers all actor positions in a single drag reflow.

**Click on message = select only (Option A, v0.9.13, final).** Kind changes via Properties panel. dblclick cycles direction. The 220ms debounce timer was the root cause of every message interaction bug in v0.9.10–0.9.12. Removing click-to-cycle-kind eliminated the entire class.

**`message:updated` calls `render()` directly (v0.9.13).** Before v0.9.13 it used `setTimeout(render, 0)` to prevent in-flight click events from hitting a freshly rebuilt DOM element. That deferral was only needed to support the click-to-cycle-kind path, which was deleted in v0.9.13. All `*:updated` listeners are now synchronous — do not re-introduce deferral without a concrete in-flight-event repro.

**`save` fires on every `render()`.** Save-on-render, not save-on-dispatch. `render()` is already the single convergence point. Cost is one `JSON.stringify` per render — acceptable at current diagram sizes. Do not pre-optimize.

**Fragment layer order.** lifelines → fragments → messages → notes → actors. Selected fragment renders last within its layer.

**`_ref` is a frozen copy.** `_wrapSelected` freezes `{ ...obj }`, not `obj` itself.

**Redo stack as captured snapshots, not log replay.** Log replay would re-execute handlers with side effects. Snapshot capture is O(1) per step and deterministic.

**`pushSnapshot()` as the single redo-invalidation point.** Every new undoable mutation calls `pushSnapshot()`, which clears `_redoStack`. The "new action kills forward history" rule is enforced once, not scattered.

**`canUndo` / `canRedo` as store getters.** Reads `_snapshots.length` / `_redoStack.length` directly — the actual source of truth. Log scans diverge from stack depth after the 50-step limit.

**`_source` as a payload convention, not a store field.** `LOAD_DIAGRAM` accepts `payload._source` as a caller hint, forwards it in `diagram:loaded`, but never writes it to `store.state`. Listeners branch on `p?.source` for toast wording.

**Zoom via `<g id="viewport">` transform.** SVG group transform keeps coordinate space intact — drag math divides by `_zoom` once and is correct. CSS scaling on the element would require compensating for `getBoundingClientRect()` returning scaled values.

**`tbtn-group` divs as the wrap unit.** Related controls stay together on one row when viewport narrows. `border-left` on `.tbtn-group + .tbtn-group` replaces `top-sep` divs. Full-width zero-height spacer forces row break without media queries.

**Visitor pattern for property panel.** Single registration point for `html`, `apply`, `announce`, and `afterRender`. Eliminates the open/closed violation of a 4× type switch split across `renderProperties` and `applyPropertyChange`.

## Architecture Notes and Known Limitations

**`dispatch()` error boundary covers log integrity, not state integrity.** The try/catch ensures a failed handler doesn't leave a phantom log entry. It does not roll back a partial state mutation. Acceptable at current scale.

**Full SVG rebuild on every `render()`.** O(n) DOM writes for all elements per call. `requestRender()` caps drag renders at 60fps (the batching problem). The underlying per-render cost remains. Targeted SVG redraws (updating only the moved element's `transform`) are the correct next optimization, listed in What's Next.

**`uiState.selected._ref` is the most dangerous pointer in the codebase.** The visitor `apply(s)` methods all dispatch correctly — but one future developer writing `s._ref.label = ...` instead of `dispatch(UPDATE_ACTOR, ...)` will introduce a silent undo bug with no visible error. `Object.freeze` on `_ref` enforces the read-only contract at the UI layer.

**Undo history intentionally not persisted.** `localStorage` saves `store.state` only. `LOAD_DIAGRAM` clears the snapshot stack as a new baseline. This is documented in the INIT boot sequence comment and is expected behavior.

**Collaborative editing is not supported.** The action log is the right foundation, but the store is a single shared mutable object. CRDT-readiness would require `{ ...action, originId, vectorClock }` on the log shape. Not a near-term concern.

**`_msgClickTimer` must be at module scope, not per-closure.** Any debounce timer that must be cancellable across render cycles belongs at module scope. Per-closure timers get a fresh `null` on every `render()` call, making the dblclick handler clear the wrong reference. This was the original bug before v0.9.13 removed the mechanism entirely — documented here as a pattern warning.

**Debug log pre-captures mutable values.** `ref.direction` read after `store.dispatch()` logs the post-mutation value. Capture `const wasDir = ref.direction` before dispatch for any "before" state recording. Store objects are mutated in place.

**`_resizeHandle` must be cleared on normal move-drag start.** `dragging._resizeHandle = null` in the mousedown guard for move-drag prevents a prior resize session from bleeding into a subsequent move.

**`place-actor` mode ghost is `uiState`, not `store.state`.** `uiState.placeGhostX` drives the ghost render — it doesn't survive a render cycle reset, which is correct. Ghost is rendered at the end of the actors layer so it sits above all other actors.
