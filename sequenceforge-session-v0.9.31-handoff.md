## Release checklist (every version bump)

Every step happens in the same commit. No separate steps. No extra decisions.

1. Gate green: http://localhost:3799/test
2. Tell Claude the version number
3. Claude does all of the following:
   a. Bump version in sequence-builder.html:
      - Header comment:  Version: X.Y.Z
      - Debug console:   SequenceForge vX.Y.Z -- debug console ready
   b. Review tour steps -- update any step that references changed UI
      Tour steps live in sequence-builder.html inside the STEPS array in the sf-tour-script block.
      Each step has: target, title, body, position.
      Update body text if a feature changed. Add a step if a major feature shipped. Remove a step if removed.
   c. Call POST http://localhost:3799/snapshot?v=X.Y.Z
      Creates releases/vX.Y.Z/sequence-builder.html automatically
   d. git add -A
   e. git commit -m "vX.Y.Z - plain ASCII description"
   f. git push origin main

Latest link always tracks main.
Stable link is pinned: releases/vX.Y.Z/sequence-builder.html

## Tour system

The tour is a self-contained overlay in sequence-builder.html.
It has no external dependencies and no build step of its own.

Location in HTML:
  CSS:    <style id="sf-tour-css">  (in <head>)
  HTML:   (before </body>)
  Script: <script id="sf-tour-script"> (before </body>)

Tour steps array (inside sf-tour-script):
  var STEPS = [
    { target, title, body, position },
    ...
  ]

  target:   CSS selector or element id (e.g. '#btn-add-actor', '#sidebar')
  title:    Short name shown in card header
  body:     HTML string -- use <b> for accent, <kbd> for key labels
  position: 'right' | 'left' | 'bottom' | 'top' | 'center'
            center = full-screen dim, no spotlight, card centered

User preference: localStorage key: sf-tour-done
  If set:   tour does not auto-launch on startup
  If cleared: tour launches on next page load

Settings modal:
  "Quick Tour" checkbox syncs to this key
  "Launch tour now" button opens tour and closes settings

Re-launch: click the ? button in the toolbar, or via Settings modal

When to update the tour (at release time):
  New element type added       -> add a step
  Button renamed or moved      -> update that step's target + body
  Feature removed              -> remove its step
  Major workflow changed       -> rewrite affected step body

Keep steps short -- 1-2 sentences each. Users are explorers, not readers.

## Architecture

```
User event
    |
    v
store.dispatch(action)   <- ONLY mutation entry point
    |
    +-- emit(event, payload)
    |
    v
store.on() listeners
    |
    +-- setSelected(obj, type, doRender)  <- ONLY selection entry point
    +-- render() or requestRender()
            |
            v
    renderElement(el, type, layer, ...)  <- ONLY scaffolding
            |
            +-- drawFn(g)   type-specific content
            +-- clickFn(e)  type-specific click
```

Invariant: store never reads uiState. store never calls render().

## State model

```
store.state -- serializable, persistable
{
  actors:    [{ id, x, label, type, emoji? }],
  messages:  [{ id, fromId, toId, label, kind, direction, y, protocol?, port?, auth?, dataClass? }],
             fromId/toId always null or valid actor id, never undefined
  notes:     [{ id, x, y, text }],
  fragments: [{ id, x, y, w, h, kind, cond }],
  nextId:    Number
}

uiState -- never persisted
{
  selected:     null | { ...obj, _type, _ref, _preview? },
  mode:         'select' | 'connect' | 'place-actor',
  connectFrom:  null | actorId,
  format:       'plantuml' | 'mermaid',
  placeGhostX:  null | Number
}
```

## Six things most likely to confuse a fresh AI instance

1. Never edit @@STORE-START...@@STORE-END in HTML -- generated by build.js
2. _wrapSelected freezes a COPY: Object.freeze({ ...obj }) not obj itself
3. render() is immediate, requestRender() is rAF-batched -- never swap them
4. state = _store.state is a live alias -- writing directly bypasses undo
5. REFLOW_ACTORS for batch moves -- one undoable step, not N
6. _parseUML returns { actors, messages, warnings } -- warnings always present

Plain ASCII only. No em-dashes, no Unicode. They corrupt on terminal copy-paste.
Good: `v0.9.31 - fix palette icons in collapsed mode adding to canvas`
Bad:  `v0.9.31 -- fix palette icons` (em-dash corrupts)

## proto2prod rules (carry forward)

- Validates before optimizing (breadth-first then depth)
- Single-file/zero-dep prototypes to learn before adding infrastructure
- Documents for two audiences: human + fresh AI instance
- Edits surgically -- verifies targets, checks side-effects
- Code should never bury run-time knowledge: visible errors over graceful degradation
- Memory entries refactor over time -- transferable patterns over project-specific detail

UI rule: any element can be added at any time -- no precondition guards
(e.g. no "need 2 actors before adding a message")
Association between elements done via Properties or drag, never via implicit arm-and-fire mechanics.

SequenceForge release flow (v0.9.29+):
1. Gate green at /test
2. User tells Claude the version number
3. Claude bumps version strings, calls POST /snapshot?v=X.Y.Z, includes everything in one commit
